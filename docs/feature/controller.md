# Controller 설계 문서

## 1. 역할과 책임

Controller는 사용자 입력을 읽고, Model을 갱신하며, View 렌더링을 지시하는 계층이다. `../CLAUDE.md`의 원칙에 따라:

- 콘솔로부터 입력을 읽는 유일한 계층이다.
- 메뉴 내비게이션 루프(메인 메뉴 → 서브메뉴 → 뒤로가기/종료)를 소유한다.
- 임의 깊이의 중첩 메뉴를 지원해야 하며, 단일 계층의 flat switch로 구현하지 않는다.
- 도메인 로직(예: 재고 부족 시 생산 라인 등록)은 이 저장소가 아니라 `SampleOrderSystem`에서 이 Controller 계약을 구현/확장하여 처리한다. 이 문서는 **Controller가 지켜야 할 공통 동작 규칙과 예외 처리 정책**을 정의한다.

## 2. 핵심 구성요소

### 2.1 `IController` / `MenuNode`
- 모든 화면(메뉴/서브메뉴/실행형 액션)은 공통 인터페이스를 구현한다.
- 각 `MenuNode`는 다음을 가진다: 표시 라벨, 실행 시 호출되는 핸들러(다음 `MenuNode`로 전이하거나 액션을 수행), 부모 메뉴로 되돌아갈 수 있는 참조.
- 메뉴 트리는 컴포지트 패턴으로 구성하여, 새로운 메뉴/서브메뉴 추가 시 기존 트리 순회 로직을 변경하지 않아도 되게 한다(개방-폐쇄 원칙).

### 2.2 `NavigationLoop`
- 현재 `MenuNode`의 View를 렌더링 → 입력 대기 → 유효성 검사 → 선택된 액션 실행 → 다음 상태(같은 메뉴 유지/하위 메뉴 진입/상위 메뉴 복귀/종료)로 전이하는 루프를 관리한다.
- 종료 조건: 최상위 메뉴에서 종료(`0`)를 선택했을 때만 루프가 끝난다.

### 2.3 `InputReader`
- 표준입력에서 한 줄을 읽고 트림(trim)하는 등 공통 입력 전처리를 담당하는 유틸리티.
- 실제 파싱(정수 변환, 범위 체크 등)은 각 액션 핸들러가 아래 규칙에 따라 처리한다.

## 3. 동작 규칙

1. **번호 선택 메뉴**: 각 메뉴는 선택 가능한 항목에 정수 번호를 매핑한다. `0`은 관례적으로 "뒤로가기"(서브메뉴) 또는 "종료"(메인 메뉴)에 예약한다.
2. **입력 전 요약 확인이 필요한 액션**(예: 예약 접수, 승인/거절, 출고 처리 등 상태를 변경하는 액션)은 실행 전 `ConfirmView`를 통해 Y/N 확인을 받는 것을 기본 규칙으로 한다. 단, 이 규칙 자체는 일반 정책이며 어떤 액션이 "상태 변경 액션"에 해당하는지는 도메인 계층(`SampleOrderSystem`)이 결정한다.
3. **되돌아가기(N/0)는 항상 부작용 없이 취소**되어야 한다 — 확인 프롬프트에서 `N`을 선택하면 Model에 어떤 변경도 가하지 않는다.
4. **연속 실행 화면**(예: 모니터링, 생산 라인 조회)은 별도 액션 없이 조회 후 자동으로 메뉴로 복귀하거나, 갱신 후 같은 화면에 머무를 수 있다 — 구체 정책은 해당 화면을 구현하는 도메인 계층에서 결정하되, Controller 계약은 "조회 전용 화면"과 "상태 변경 화면"을 구분할 수 있는 타입(예: `ReadOnlyMenuNode` vs `ActionMenuNode`)을 제공한다.

## 4. 예외/오류 처리 규칙

| 상황 | 처리 규칙 |
|---|---|
| 메뉴에 존재하지 않는 번호 입력 | 오류 메시지(`MessageView`)를 출력하고 같은 메뉴에서 재입력을 요구한다. 예외를 던지거나 프로그램을 종료하지 않는다. |
| 숫자가 아닌 입력 | 파싱 실패로 간주하여 위와 동일하게 재입력을 요구한다. |
| 빈 입력(Enter만 입력) | 기본값이 정의되어 있지 않은 한 재입력을 요구한다. |
| 확인 프롬프트에서 Y/N 외 입력 | 유효한 응답(Y 또는 N)을 받을 때까지 재요청한다. |
| Model 계층에서 유효하지 않은 상태 전이를 거부한 경우 | Controller는 이를 예외적 흐름이 아닌 "정상적으로 실패할 수 있는 결과"로 처리하여, `MessageView`로 실패 사유를 안내하고 메뉴로 복귀한다(프로그램 크래시로 이어지지 않는다). |
| 예기치 않은 런타임 예외 | 최상위(메인 루프)에서 포착하여 오류 메시지를 출력하고 메인 메뉴로 복귀시킨다. 애플리케이션이 예외로 인해 비정상 종료되지 않도록 한다. |

## 5. Controller가 하지 않는 것 (Non-goals)

- 화면 레이아웃/색상 등 출력 포맷 결정 (View 책임)
- 데이터 저장/불러오기 구현 (`DataPersistence` 책임 — Controller는 Model이 노출하는 계약을 통해서만 접근)
- 도메인 특화 상태 전이 규칙(RESERVED → CONFIRMED/PRODUCING 등)의 구체적 정의 — 이는 `SampleOrderSystem`이 Model 계층에 구현하고, Controller는 그 결과(성공/실패)만 위 표의 규칙에 따라 처리한다.
