# View 설계 문서

## 1. 역할과 책임

View는 콘솔 화면 출력만을 담당한다. `../CLAUDE.md`의 원칙에 따라:

- View는 상태를 소유하지 않는다 — Controller/Model로부터 표시할 데이터를 전달받아 그릴 뿐이다.
- View는 사용자 입력을 직접 읽지 않는다 (입력 읽기는 Controller의 책임이며, View는 "무엇을 입력하라"는 프롬프트 문구만 출력한다).
- 화면 레이아웃(테두리, 표 형식, 색상 여부 등)은 `../ref/requirements.pdf`에 반복적으로 명시된 것처럼 **자유롭게 결정 가능**하며, PDF의 예시 화면은 참고용일 뿐 강제 사양이 아니다.

## 2. 참고: 요구사항 문서의 예시 화면에서 도출한 공통 UI 패턴

`requirements.pdf`의 각 메뉴별 "예시 UI 화면"(메인 메뉴, 시료 관리, 시료 주문, 주문 승인/거절, 모니터링, 생산라인, 출고 처리)을 분석하면, 화면마다 반복되는 패턴이 존재한다. 이 패턴들을 일반화하여 재사용 가능한 View 구성요소로 정의한다.

| 관찰된 패턴 | 예시 (PDF 근거) | 일반화된 View 구성요소 |
|---|---|---|
| 번호 선택형 메뉴 | `[1] 시료 관리  [2] 시료 주문 ... [0] 종료` | `MenuView` — 항목 목록과 선택 프롬프트 렌더링 |
| 표 형태 목록 | 시료 목록(ID/이름/평균생산시간/수율/재고), 주문 목록(주문번호/고객/시료/수량/상태) | `TableView` — 컬럼 헤더 + 행 렌더링 |
| 페이지네이션 | `...외 7종  [N] 다음페이지` | `TableView`의 페이지 이동 옵션 |
| 상태 배지 | `RESERVED`, `CONFIRMED`, `PRODUCING`, `RELEASE`, `REJECTED` 등 | `BadgeView` — 임의의 라벨 문자열을 강조 표기 (색상/괄호 등은 구현 자유) |
| 입력 확인/요약 | "입력 내용 확인 → [Y] 예약 접수 [N] 취소" | `ConfirmView` — 요약 정보 표시 + Y/N 프롬프트 |
| 진행률/잔여율 표시 | 생산 진행률 `72%`, 재고 잔여율 `80%` 게이지 | `ProgressBarView` — 0~100% 값을 막대/퍼센트로 표시 |
| 결과/완료 메시지 | "예약 접수 완료.", "출고 처리 완료." | `MessageView` — 성공/실패/안내 메시지 스타일 구분 |
| 헤더/타임스탬프 | `[4] 모니터링   2026-04-16 09:32:15` | `HeaderView` — 화면 제목 + 부가 정보(시각 등) 표시 |

**주의**: 위 표의 "예시"는 실제 도메인 데이터(시료명, 주문 상태 등)를 포함하지만, 이는 어디까지나 요구사항 문서의 사례일 뿐이다. `ConsoleMVC`가 실제로 구현하는 `TableView`, `BadgeView` 등은 컬럼명/라벨 값을 매개변수로 받는 **범용 컴포넌트**여야 하며, 시료/주문 관련 문자열을 하드코딩하지 않는다.

## 3. 공통 View 구성요소 정의

### 3.1 `HeaderView`
- 화면 제목과 부가 정보(타임스탬프 등, 값은 외부에서 주입)를 상단에 렌더링

### 3.2 `MenuView`
- `{ 번호, 라벨 }` 목록을 입력받아 번호 선택형 메뉴를 렌더링
- 항상 "뒤로가기/종료" 항목(번호 `0`)을 포함할 수 있는 규칙을 지원 (실제 라벨은 호출부가 결정)

### 3.3 `TableView`
- 임의의 컬럼 정의(헤더 이름 목록)와 행 데이터(문자열 셀 목록)를 받아 표 형태로 렌더링
- 페이지 크기를 초과하는 데이터에 대해 "다음 페이지" 안내를 표시하는 옵션 지원

### 3.4 `BadgeView`
- 임의의 상태 라벨 문자열을 강조된 형태로 감싸 출력 (예: 대괄호, 색상 등 — 구현 자유)
- 라벨과 스타일(정상/경고/오류 등 카테고리)만 매개변수로 받고, 실제 상태 값 목록은 알지 못한다

### 3.5 `ConfirmView`
- 요약 정보(키-값 목록)를 표시한 뒤 Y/N 선택을 요청하는 프롬프트를 렌더링

### 3.6 `ProgressBarView`
- 0.0 ~ 1.0 사이의 비율 값을 받아 막대 그래프 + 퍼센트 텍스트로 렌더링

### 3.7 `MessageView`
- 성공/실패/안내 메시지를 구분된 스타일로 출력 (예: 접두어, 색상 등)

## 4. 입력 프롬프트와의 관계

View는 "선택 > _" 같은 입력 프롬프트 **문구**만 출력하고, 실제 표준입력 읽기는 Controller가 수행한다 (`controller.md` 참고). 이는 View가 입력 채널(콘솔, 향후 GUI/테스트용 스텁 등)에 대해 알 필요가 없도록 하기 위함이며, 향후 View 구현을 교체(예: 색상 지원 콘솔, 로그 기반 테스트 더블)하더라도 Controller 로직이 영향받지 않게 한다.

## 5. View가 하지 않는 것 (Non-goals)

- 상태 저장 (Model 책임)
- 입력 읽기/파싱/검증 (Controller 책임)
- 도메인 문자열(시료명, 주문 상태 값 등) 하드코딩
