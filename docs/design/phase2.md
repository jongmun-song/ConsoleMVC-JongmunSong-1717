# Phase 2 — View 계층 구현

## 목표

`docs/feature/view.md`에 정의된 공통 View 컴포넌트를 구현한다. View는 Model 상태를 직접 참조하지 않고, 호출부(Controller)로부터 전달받은 데이터만 렌더링한다.

## 근거 문서

- `../feature/view.md` 전체

## 작업 항목

1. **`HeaderView`**: 화면 제목 + 부가 정보(타임스탬프 등, 외부 주입) 렌더링
2. **`MenuView`**: `{번호, 라벨}` 목록 → 번호 선택형 메뉴 렌더링
3. **`TableView`**: 임의 컬럼 헤더 + 행 데이터 → 표 렌더링, 페이지 크기 초과 시 "다음 페이지" 안내 옵션
4. **`BadgeView`**: 임의의 라벨 문자열 + 스타일 카테고리(정상/경고/오류) → 강조 표기
5. **`ConfirmView`**: 키-값 요약 정보 표시 + Y/N 프롬프트 문구 렌더링
6. **`ProgressBarView`**: 0.0~1.0 비율 → 막대 그래프 + 퍼센트 텍스트
7. **`MessageView`**: 성공/실패/안내 메시지를 구분된 스타일로 출력

각 컴포넌트는 순수 함수(또는 상태 없는 클래스)에 가깝게 작성한다 — 동일한 입력에 대해 항상 동일한 출력 문자열을 생성해야 테스트가 쉬워진다.

8. **PoC 예시 도메인 화면 검증**: `Example/View/`에 시료 목록/등록, 주문 접수/승인/거절, 모니터링, 생산라인 현황, 출고 처리 화면을 위 1~7번 범용 컴포넌트를 조합하여 구현한다(`../feature/view.md` §7 참고). Phase 1에서 만든 `Example/Model/`의 예시 도메인 데이터를 입력으로 사용해 렌더링 결과를 확인한다. 이 항목은 화면 렌더링이 범용 컴포넌트만으로 실제 도메인 형태를 표현할 수 있는지 검증하는 데모이며, 코어(`View/`)는 수정하지 않는다.

## 확장성 관련 참고

View 자체는 모니터링 확장과 직접 관련이 없다(모니터링 확장 지점은 Model 계층/Phase 1에 있음). 다만 `TableView`/`BadgeView`처럼 상태·수치를 표 형태로 보여주는 컴포넌트는, 추후 `DataMonitor`가 유사한 콘솔 UI가 필요할 경우 그대로 재사용할 수 있도록 도메인 문자열을 하드코딩하지 않고 매개변수화된 형태를 유지한다.

## 완료 기준

- 위 7개 컴포넌트가 `View/` 디렉터리에 구현되어 있고 컴파일된다.
- 어떤 컴포넌트도 `std::cin` 등 입력을 읽지 않는다(입력은 Controller 책임).
- 어떤 컴포넌트도 도메인 특화 문자열(시료명, 주문 상태 값 등)을 상수로 갖지 않는다.
- (8번 항목) `Example/View/`의 예시 도메인 화면이 위 범용 컴포넌트만을 조합하여 구현되어 있고, 예시 도메인 데이터로 렌더링 결과를 확인할 수 있다.

## 리뷰 포인트 (code-reviewer)

- View가 상태를 들고 있지 않은지(호출마다 필요한 데이터를 인자로 받는지)
- 입력 프롬프트 "문구" 출력과 실제 입력 "읽기"가 섞여 있지 않은지
- 함수/클래스 이름이 렌더링 대상과 목적을 명확히 드러내는지

## 테스트 포인트 (tester)

- 각 View 컴포넌트에 대해 입력 데이터 → 출력 문자열이 기대한 형식을 포함하는지(예: `TableView`가 모든 행을 순서대로 포함하는지, `ProgressBarView`가 0%/100%/중간값에서 올바른 텍스트를 생성하는지)
- 페이지네이션 경계값(정확히 페이지 크기만큼일 때, 한 개 초과일 때)
- `BadgeView`/`MessageView`의 스타일 카테고리별 출력 분기
- (8번 항목) `Example/View/`의 예시 화면이 `Example/Model/` 데이터를 받아 기대한 출력을 생성하는지

## Out of scope

- 실제 입력 읽기/파싱 (Phase 3)
- 예시 도메인 화면과 실제 메뉴 트리(Controller)의 연결 — 화면 렌더링 자체(8번 항목)만 이 Phase의 범위이며, 메뉴 내비게이션과의 통합은 Phase 3/4에서 다룬다
